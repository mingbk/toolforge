🎲 随机字符串生成：真随机 vs 伪随机

从原理到实践，理解加密级随机数的真正含义

为什么需要"真"随机？

写一个随机字符串生成器很简单——但写一个安全的随机字符串生成器，就不是随便调个 Math.random() 能搞定的了。如果你生成的随机字符串是用来做密码重置 Token、验证码、API Key —— 那攻击者能预测"随机"结果的话，就等于拿到了你系统的钥匙。

核心区别：Math.random() 是"看起来随机"的伪随机数生成器（PRNG）；crypto.getRandomValues() 是密码学安全的真随机数生成器（CSPRNG）。前者快但可预测，后者慢但不可能被预测。

Math.random()：为什么不够安全？

工作原理

Math.random() 使用的是伪随机数生成器（通常是 xorshift128+ 算法）。它从一个初始"种子"（seed）开始，通过数学公式不断生成下一个数。给定相同的种子，就会产生完全相同的随机数序列。

三个致命弱点

  • 可预测性：攻击者如果能猜出种子（比如用当前时间戳做种子），就能重现整个随机序列
  • 状态可恢复：攻击者拿到连续的几个随机数输出，就可以反推内部状态，预测未来输出
  • 设计目的不同：V8 引擎的 Math.random() 优化目标是速度和分布均匀性，不是安全性
// ❌ 不要用 Math.random() 生成安全令牌！
function unsafeToken(len) {
  const chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789';
  let result = '';
  for (let i = 0; i < len; i++) {
    result += chars[Math.floor(Math.random() * chars.length)];
  }
  return result;
}

crypto.getRandomValues()：密码学安全随机

工作原理

crypto.getRandomValues() 是 Web Crypto API 提供的密码学安全随机数生成器。它从操作系统底层获取真正的熵源——鼠标移动、键盘时序、硬盘中断、网络包到达时间等硬件级别的随机事件，是不可预测的。

为什么它安全？

  • 真熵源：不是数学公式算出来的，而是从物理世界采集的随机性
  • 不可逆推：拿到输出也无法反推内部状态或预测未来值
  • 不可被播种：没有"种子"概念，无法复现
  • 浏览器原生支持：所有现代浏览器都内置了，无需任何依赖
// ✅ 使用 crypto.getRandomValues() 生成安全令牌
function safeToken(len) {
  const chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789';
  const arr = new Uint32Array(len);
  crypto.getRandomValues(arr);
  let result = '';
  for (let i = 0; i < len; i++) {
    result += chars[arr[i] % chars.length];
  }
  return result;
}

实际场景：你什么时候需要加密级随机？

  • 🔑 生成 API Key / Access Token：随便用 Math.random() 等于把钥匙送给攻击者
  • 📧 密码重置链接：Token 可预测 = 任意用户账号可被接管
  • 🎫 优惠码/邀请码：如果能猜出规律，优惠券会被薅光
  • 🎮 游戏内的"随机"掉落（不重要）：用 Math.random() 完全没问题
  • 🆔 唯一ID生成：如果你用 UUID v4，它底层已经用了 crypto

取模偏差（Modulo Bias）问题

即使用了 crypto.getRandomValues()，用取模 arr[i] % charset.length 来选字符，理论上存在取模偏差。比如 Uint32 范围是 0~4294967295，如果字符集长度不能整除这个范围，某些字符出现的概率就会略高。

实际上：当字符集长度远小于 2³² 时，偏差微乎其微（< 0.0000001%），对绝大多数场景没有实际影响。如果要求极致均匀，可以改用拒绝采样法（rejection sampling），但对日常使用来说，取模已经足够。

常见问题

Q: crypto.getRandomValues 会耗尽随机数吗？

操作系统维护了一个"熵池"（entropy pool），持续从硬件事件中收集随机性。普通应用几乎不可能耗尽它。浏览器内一次调用最多生成 65536 字节，完全够用。如果需要大量随机数，可以分批调用。

Q: Node.js 里有 crypto.getRandomValues 吗？

有！Node.js 的 crypto 模块提供了 crypto.randomBytes()（传统 API）和 crypto.getRandomValues()（Web 兼容 API）。两者都使用操作系统的 CSPRNG，是完全安全的。

Q: Math.random() 真的那么容易被破解吗？

对于 V8 引擎的 xorshift128+ 实现，攻击者只需要观察到大约 3-4 个连续的 Math.random() 输出值（64位浮点数），就能通过 Z3 求解器反推出内部状态。2018 年已经有公开的 PoC 攻击代码。
